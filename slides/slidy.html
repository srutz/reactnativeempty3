Javascript ES6 (Arrow Functions)

  • Arrow Functions sind eine kürzere Schreibweise für Funktionen. Der Hauptunterschied ist, dass sie kein eigenes this haben.
function add(a, b) { return a + b}
const add = (a, b) => { return a + b }
const add = (a, b) => a + b

Javascript ES6 (Module)

  • Module sind eine Möglichkeit, Code in mehrere Dateien aufzuteilen. Die Datein importieren und exportieren Funktionen und Konstanten.

  • Arten von Exports

    • Default Export
    • Named Export

  • Arten von Imports

    • Default Import
    • Named Import
// export.js
export const add = (a, b) => a + b
export default add
// import.js
import add, {add} from './export.js'
  • Eine Umbenennung ist ebenfalls möglich
import {add as sum} from './export.js'

Javascript ES6 (Promises)

  • Promises sind ein Konzept, um asynchrone Operationen zu handhaben. Sie sind ein Container für einen Wert, der jetzt oder später verfügbar sein kann.
const delayed = new Promise((resolve, reject) => {
    setTimeout(() => {
        resolve('Success')
    }, 5_000) // resolved nach 5000 Millisekunden
})
  • Auf ein Promise kann man mit await warten. Dafür muss der Aufruf in einer async Funktion stehen.
const result = await delayed(5_000)
  • Falls man sich nicht in einer async Funktion befindet, kann man mit dem IIFE Pattern (Immediately Invoked Function Expression) arbeiten.
(async () => {
    const result = await delayed(5_000)
    console.log(result)
})()

Überblick - Was ist React?

  • React ist eine JavaScript-Bibliothek, die von Facebook entwickelt wurde.
  • React wird verwendet, um Benutzeroberflächen zu erstellen.
  • React wurde 2013 veröffentlicht.
  • React ist eine Open-Source-Bibliothek.

Das Grundprinzip von React

  • React verwendet ein Konzept namens Virtual DOM.
  • Es erstellt eine virtuelle Repräsentation des DOM.
  • Bei Änderungen wird der virtuelle DOM aktualisiert.
  • Der virtuelle DOM wird dann mit dem tatsächlichen DOM verglichen.
  • Achtung: der virtuelle DOM (VDOM) ist eine vereinfachte Kopie des tatsächlichen DOM und wird nur intern von React verwendet.

Der Abgleich von virtuellem und tatsächlichem DOM

  • React vergleicht den virtuellen DOM mit dem tatsächlichen
  • React erkennt, welche Teile des DOM aktualisiert werden müssen
  • React aktualisiert nur die Teile, die sich geändert haben
  • Dafür vergleicht React den aktuellen Zustand mit dem vorherigen Zustand
  • React aktualisiert nur die Teile, die sich geändert haben

React-Komponenten

  • React-Anwendungen bestehen aus Komponenten
  • Komponenten sind wiederverwendbare Code-Blöcke
  • Komponenten können in andere Komponenten eingebettet werden
  • Komponenten können Daten an andere Komponenten übergeben (Props)

JSX (JavaScript XML)

  • JSX ist eine Erweiterung von JavaScript
  • JSX steht für JavaScript XML
  • JSX ermöglicht das Schreiben von HTML-Code in JavaScript
  • JSX wird von React verwendet, um Komponenten zu erstellen
  • JSX kann auch durch Aufrufe von React.createElement() ersetzt werden, wodurch es aber keinen Performance-Vorteil gibt

React und ReactNative

  • Eine klassische Miniale React-Anwendung verwendet die Packages react und react-dom.
npm install react react-dom
  • Die React Native Anwendung verwendet die Packages react und react-native.
npm install react react-native
  • Man sieht dass beide Anwendungen die React-Bibliothek verwenden, aber unterschiedliche DOM-Implementierungen.

Was sind React Native Komponenten?

  • Klassische React Anwendungen bestehen aus Komponenten die HTML Elemente verwenden.
  • React Native Anwendungen bestehen aus Komponenten die nativen UI Elemente verwenden.
  • Diese nativen UI Elemente sind für iOS und Android spezifisch aber werden von React Native abstrahiert.
  • Die wichtigsten Elemente sind:
    • View, Text, Image, TextInput
    • ScrollView, FlatList, SectionList
    • Button, TouchableHighlight, TouchableOpacity
    • ActivityIndicator, Alert, Modal
    • Picker, Slider, Switch, WebView
  • React Native verwendet keine HTML Elemente.
  • expo-dom ist ein Projekt das React-DOM Komponenten in Reactnative erlaubt (Stand Oktober 2024: noch in der Entwicklung)

Unterschiede zwischen React und React Native

  • React verwendet HTML Elemente
  • React Native verwendet native UI Elemente
  • React verwendet CSS
  • React Native verwendet Flexbox
  • React verwendet Webpack/Vite/Turbopack
  • React Native verwendet Metro Bundler
  • React verwendet Browser APIs
  • React Native verwendet nativen APIs
  • React Native läuft nicht auf dem Server (kein SSR, SSG)

Gemeinsamkeiten zwischen React und React Native

  • Beide verwenden JSX
  • Beide verwenden Komponenten
  • Beide verwenden Props und State
  • Beide verwenden Lifecycle-Methoden
  • Beide verwenden Hooks
  • Beide verwenden Context API
  • Beide verwenden Redux, Zustand oder andere State-Management-Lösungen
  • Beide verwenden Router (React-Router, React-Navigation, Expo-Router)

Die Programmierung ist also weitesgehend identisch, nur die UI-Elemente sind unterschiedlich.

Unterschiede beim Styling

  • React verwendet CSS
  • React Native verwendet Flexbox
  • React Native verwendet keine Klassen
  • React Native verwendet Inline-Styles und StyleSheet API
  • Tailwind CSS kann nicht in React Native verwendet werden
  • Nativewind ist ein Projekt das Tailwind CSS in React Native annähernd nachbildet
  • Media Queries können in React Native nicht verwendet werden
  • Das Styling in React Native ist eingeschränkt im Vergleich zu React.

NPM und Node_Modules

  • Generell werden abhängigkeiten in der Datei package.json festgehalten.
  • Projekte sind oftmals so aufgesetzt, das sowohl die Abhängigkeiten als auch die Entwicklungsabhängigkeiten in der Datei package.json festgehalten werden.
  • Solche Projekte werden dann nach dem Herunterladen der Sourcen mit npm install initialisiert.
  • Alle Abhängigkeiten werden dann in einem Verzeichnis node_modules abgelegt und man ist nicht von der Versionen der global installierten Pakete abhängig. (Empfohlen)
  • Die node_modules können sehr groß werden, PNPM ist eine Alternative, die Duplikate mittels Symlinks vermeidet und ein zentrales Repository für alle Projekte verwendet.
  • Eine weitere Alternative ist Yarn, das von Facebook entwickelt wurde und eine schnellere Installation von Paketen ermöglicht. Yarn installiert parallel und verwendet ebenfalls einen zentralen Cache.

JSX (Javascript XML) als Templatesprache

  • JSX ist eine Erweiterung von JavaScript
  • JSX steht für JavaScript XML
  • JSX ermöglicht das Schreiben von HTML-Code in JavaScript

Unterschiede zu HTML

  • JSX ist nicht HTML
  • Tags müssen geschlossen werden
  • Attribute müssen in Anführungszeichen stehen
  • Klassennamen müssen als className geschrieben werden
  • Inline-Styles müssen als Objekt übergeben werden
  • Inline-Styles müssen in geschweiften Klammern stehen
  • Werte die keine Zeichenketten sind müssen in geschweiften Klammern stehen

Beispiel

import React from 'react';
import { Text } from 'react-native';
export function BigText() {
    return <Text style={{fontSize: 20}}>Hallo Welt</Text>;
}

Javascript in JSX

  • Javascript kann in JSX verwendet werden.
  • Die Haupteinschränkung ist, dass nur Javascript-Ausdrücke und KEINE Statements verwendet werden können.
  • Um Ausdrücke zu verwenden, müssen diese in geschweifte Klammern geschrieben werden.

Beispiel

import React from 'react';
import { Text } from 'react-native';
export function BigText({message} : {message: string}) {
    return <Text style={{fontSize: 20}}>{message}</Text>;
}

Children in JSX

  • Kinder-Komponenten können in JSX übergeben werden.
  • Kinder-Komponenten werden als children übergeben.
  • Kinder-Komponenten können beliebig viele Kinder haben.

Beispiel

import React from 'react';
import { Text } from 'react-native';
export function BigText({children} : {children: string}) {
    return <Text style={{fontSize: 20}}>{children}</Text>;
}

Props in JSX

  • Props sind Attribute die an Komponenten übergeben werden. Quasi vergleichbar mit HTML-Attributen.
  • Props werden als Objekt übergeben.
  • Props können beliebig viele Attribute haben.

Beispiel

import React from 'react';
import { Text } from 'react-native';
export function BigText({style, children} : {style: object, children: string}) {
    return <Text style={style}>{children}</Text>;
}

In dieser Syntax wird ein Objekt mit zwei Attributen übergeben. Das Attribut style ist ein Objekt und das Attribut children ist eine Zeichenkette. Es wird direkt in der Funktion sowohl die Typisierung als auch die Dekonstruktion der Props vorgenommen.

Alternative Syntax mit Zwischenschritten

import React from 'react';
import { Text } from 'react-native';
type BigTextProps = {
    style: TextStyle,
    children: string
}
export function BigText(props : BigTextProps) {
    const {style, children} = props;
    return <Text style={style}>{children}</Text>;
}

In dieser Syntax wird zuerst ein Typ BigTextProps definiert, der die Typisierung der Props enthält. Dann wird die Funktion BigText definiert, die die Props als Parameter erhält. In der Funktion wird dann die Dekonstruktion der Props vorgenommen.

Default Werte for Props

import React from 'react';
import { Text } from 'react-native';
type BigTextProps = {
    style: TextStyle,
    children: string
}
export function BigText(props : BigTextProps) {
    const {style = {fontSize: 20}, children} = props
    return <Text style={style}>{children}</Text>;
}

Fallunterscheidung in JSX

  • Fallunterscheidungen können in JSX mit dem ternären Operator durchgeführt werden.

Beispiel

import React from 'react';
import { Text, TextStyle } from 'react-native';
type BigTextProps = {
    style: TextStyle,
    children: string
}
export function BigText(props : BigTextProps) {
    const {style = {fontSize: 20}, children} = props
    return <Text style={style}>{children.length > 10 ? children : 'Text zu kurz'}</Text>;
}

Schleifen in JSX

  • Schleifen können in JSX mit der map-Funktion durchgeführt werden.

Beispiel

import React from 'react';
import { Text, TextStyle } from 'react-native';
type BigTextProps = {
    style: TextStyle,
    children: string[]
}
export function BigText(props : BigTextProps) {
    const {style = {fontSize: 20}, children} = props
    return <Text style={style}>{children.map((child, index) => <Text key={index}>{child}</Text>)}</Text>;
}
  • While und for-Schleifen können nicht in JSX verwendet werden, da JSX nur Ausdrücke und keine Statements erlaubt.

Fragments in JSX

  • Fragments sind Container, die mehrere Kinder-Komponenten enthalten können.
  • Fragments werden als leeres Tag geschrieben oder mit <React.Fragment>.

Beispiel

import React from 'react';
import { Text, TextStyle } from 'react-native';
type BigTextProps = {
    style: TextStyle,
    children: string[]
}
export function BigText(props : BigTextProps) {
    const {style = {fontSize: 20}, children} = props
    return <React.Fragment>{children.map((child, index) => <Text key={index}>{child}</Text>)}</React.Fragment>;
}

Kommentare in JSX

  • Kommentare in JSX werden in geschweiften Klammern geschrieben.
  • Sprich man wechseln von JSX in Javascript und schreibt dann den Kommentar in /* */ oder mitels // bis zum Zeilenende.
<View>
     { /* Kommentar */}
    <Text>Text</Text>
</View>

JSX-Zusammenfassung

  • JSX ist fast wie Html. Folgende wichtige Unterschiede existieren:
    • {} geschweifte Klammern erlauben direktes Einbetten von Javascript/Typescript
    • CamelCase Schreibweise für Attribute. Groß/Kleinschreibung ist NICHT egal. (onClick anstelle von onclick oder className anstelle von class)
    • className anstelle von class, htmlFor anstellen von for (Javascript Keywords)
    • Events müssen sind Javascript-Funktionen, kein beliebiger Javascriptausdruck -Styles werden als Javascript Objekte angegeben werden. (z.B: <div style={{ color: ‘red’ }}>)
    • Fallunterscheidungen und Schleifen sind möglich, aber nur als Ausdrücke, nicht als Statements

Typescript

  • Typescript ist eine Programmiersprache die auf Javascript basiert.

  • Typescript ist eine statisch typisierte Sprache.

  • Typescript ist objektiv dann sinnvoll, wenn Projekte größer werden, mehrere Entwickler beteiligt sind oder die Software längerfristig gewartet werden soll.

  • Die Typisierung ermöglicht es Fehler zur Compile-Zeit zu erkennen und nicht erst zur Laufzeit.

  • Die DX (Developer Experience) wird verbessert, da ein Editor den Sourcecode genau analysieren kann und werterhaltend transformieren kann.

Typescript Basics

  • Typescript ist eine Erweiterung von Javascript, die Typen unterstützt. Es gibt Basisdatentypen wie number, string, boolean, null, undefined, object und symbol.
  • Eigene Typen können mit type oder mit der Altnativschreibweise interface definiert werden.
type MyNumber = number
type MyString = string
interface MyPerson {
  firstname: string
  lastname: string
  email: string
}

Typescript Basics (1)

  • In Typescript kann man Typen definiteren mittels :
type MyNumber = number
type MyString = string

type MyPerson = {
  firstname: string
  lastname: string
  email: string
}
  • Ein Typ ist “nur” ein Alias für einen Datentyp. Anstelle des Namens kann auch immer der Datentyp selber subst
  • Optionale Eigenenschaften können mit ? markiert werden
type MyPerson = {
  firstname: string
  lastname: string
  email?: string  // email ist optional
}

Typescript Basics (2)

  • Typen können mit dem & Operator kombiniert werden:
type MyPerson = {
  firstname: string
  lastname: string
}
type MyAddress = {
  street: string
  city: string
}

type MyPersonWithAddress = MyPerson & MyAddress

Typescript Basics (3)

  • Typen können auch mit dem | Operator kombiniert werden. Der resultierende Typ kann dann entweder den einen oder den anderen Typ haben.
type MyValue = string | number

Typescript Basics (4)

  • An Typen können Untertypen (Subtypes) zugewiesen werden. Diese Untertypen werden auch als Generics bezeichnet.
const [person, setPerson] = useState<MyPerson>({firstname: 'Max', lastname: 'Mustermann', email: 'a@b.de'})
  • In diesem Beispiel wird der Typ MyPerson als Typ für den State person verwendet. Die Übergabe erfolgt in spitzen Klammern <>.
  • Typen können auch aus bestehenden Typen abgeleitet werden.
const [person, setPerson] = useState<MyPerson>({firstname: 'Max', lastname: 'Mustermann', email: 'a@b.de'})

React Grundlagen revisited

  • React ist eine JavaScript-Bibliothek für die Entwicklung von Benutzeroberflächen.

  • React basiert auf der Idee, dass React den gesamten Zustand der GUI ausgibt.

  • Bei Änderungen von Variablen (genauer von Zuständen) wird die GUI automatisch aktualisiert. Dazu gibt React die GUI neu aus und vergleicht die alte GUI mit der neuen GUI. Nur die Teile die sich geändert haben, werden neu gerendert.

  • React verwendet ein Konzept namens Virtual DOM als Datenstruktur für die interne Repräsentation der GUI.

  • React verwendet JSX (JavaScript XML) als Syntaxerweiterung von JavaScript. JSX ermöglicht das Schreiben von HTML-Code in JavaScript.

  • React-Anwendungen bestehen aus Komponenten. Komponenten sind wiederverwendbare Code-Blöcke. Komponenten können in andere Komponenten eingebettet werden. Komponenten können Daten an andere Komponenten übergeben (Props).

React Hooks

  • React Hooks sind Funktionen, die es ermöglichen, Zustand und andere React-Funktionen in Funktionskomponenten zu verwenden. Dabei bindet sich der Zusand von useState an die Komponente, in der der Hook verwendet wird.

  • Per Namenskonvention beginnen Hooks mit use.

  • React Hooks sind eine Alternative zu Klassenkomponenten und ermöglichen es, Zustand und Lebenszyklusmethoden in Funktionskomponenten zu verwenden.

Rules of Hooks

  • Hooks werden in Arrays an der Instanz einer Komponente verwaltet. Deswegen müssen Hooks gewisse Regeln, die sogenannten Rules of Hooks, einhalten.
    • Hooks dürfen nur in Funktionskomponenten oder in anderen Hooks verwendet werden.
    • Hooks dürfen nicht in Schleifen, Bedingungen oder verschachtelten Funktionen verwendet werden.
    • Hooks müssen immer in der gleichen Reihenfolge verwendet werden.
    • Hooks dürfen nicht in normalen JavaScript-Funktionen verwendet werden.
    • Hooks dürfen nicht in Klassenkomponenten verwendet werden.
    • Hooks dürfen nicht in Callbacks von Event-Handlern verwendet werden.
  • PS: Hooks können sehr wohl Funktionen und Werte zurückgegeben die wiederrum keinen Unterschränkungen wie den Rules of Hooks unterliegen.

Die React Hooks

  • useRef (merkt sich Werte zwischen Renderaufrufen und kann JSX Element referenzieren)

  • useState (merkt sich den Zustand einer Komponente und löst einen Re-Render aus)

  • useEffect (führt Nebeneffekte in Funktionskomponenten aus und ersetzt die Lebenszyklusmethoden von Klassenkomponenten)

  • useContext (ermöglicht den Zugriff auf den Kontext eines Vorgängers in Funktionskomponenten)

  • useReducer (ermöglicht die Verwendung von Reducern in Funktionskomponenten)

  • useCallback (ermöglicht die Verwendung von Memoization in Funktionskomponenten)

  • useMemo (ermöglicht die Verwendung von Memoization in Funktionskomponenten)

  • Einige weitere Hooks sind

    • useImperativeHandle
    • useLayoutEffect
    • useDebugValue
    • useSuspense

Text

  • Text ist eine Komponente, die Text anzeigt.
    • Die wichtigsten Eigenschaften sind style, numberOfLines, onPress, onLongPress, children.
    • style ist ein Objekt, das die Text-Stile definiert und für jede Komponente verfügbar. Es können auch Stylesheets und CSS Klassen verwendet werden.

Text und onLayout

  • onLayout ist ein Ereignis, das aufgerufen wird, nachdem die Größe des Textes berechnet wurde.
  • Die Größe des Textes kann in der onLayout-Funktion abgerufen werden.
  • Die Größe des Textes kann auch in der useLayoutEffect-Hook abgerufen werden. Der Unterschied zu useEffect ist, dass useLayoutEffect synchron ausgeführt wird, bevor der Text gerendert wird.

View

  • View ist eine Komponente, die andere Komponenten enthält. Bei Reactnative wird View als Container verwendet um Layouts zu erstellen oder auch nur um andere Komponenten zu gruppieren und eigene Komponenten zu erstellen.

  • Bei Reactnative ist die Erstellung von Layouts mit Flexbox sehr einfach. Flexbox ist ein Layout-Modell, das die Anordnung von Elementen in einer Zeile oder Spalte ermöglicht.

  • Eigene Komponenten sind oftmals Views um modifizieren die Ansicht der enthaltenen Komponenten/Children.

Styles und Layout

  • Das Layout von React Native basieren auf Flexbox.
  • Flexbox dedeutet die Anordnung der Kinder horizontal oder vertikal.
  • Einzelne Kinder können mit flex-Attributen gesteuert werden.
  • Inline-Styles und die StyleSheet API werden verwendet.
  • Alternativ kann Nativewind verwendet werden, um die Styles zu definieren. Dafür wird eine zu klassischem TailwindCSS ähnliche Syntax verwendet.
  • PS: Die Verwendung von Nativewind ist empfohlen und wird in diesem Kurs verwendet.
  • Die Layouts werden in Views und ScrollViews angewendet.

MajorAxis und CrossAxis

  • Ohne weitere Angaben ist die MajorAxis vertikal.
  • Die CrossAxis ist die Nebenachse.

Komponenten-State und Komponenten-Ereignisse

  • Komponenten haben einen State (Zustand) und Ereignisse (State und Events)
  • Status wird mittels des eingebauten Hooks useState verwaltet.
  • Der Aufruf von useState gibt ein Array zurück, das den aktuellen Status und eine Funktion zum Setzen des Status enthält.
  • Der Status ist immer an die aktuelle Komponente gebunden.
  • Aufgerufene Funktionen die useState verwenden, gehören auch zur aktuellen Komponente. Der Status gehört also nicht zu der aufgerufenen Funktion, sondern zur aktuellen Komponente.
  • Solche Funktionen heißen gemeinhin “Hook-Funktionen” und werden mit dem Präfix use benannt.

Rerendering

Der useState Hook

  • Der useState Hook wird verwendet, um den Status einer Komponente zu verwalten.
  • Generell gelten die Rules of Hooks für alle Hooks.
    • Hooks dürfen nur in Funktionskomponenten verwendet werden.
    • Hooks dürfen nicht in Schleifen, Bedingungen oder verschachtelten Funktionen verwendet werden.
    • Hooks müssen immer in der gleichen Reihenfolge aufgerufen werden.
    • Hooks dürfen nicht in regulären JavaScript-Funktionen verwendet werden.
    • Hooks dürfen nur in Funktionskomponenten oder in anderen Hooks verwendet werden.
    • Hooks dürfen nicht in Klassenkomponenten verwendet werden.
  • Der useState Hook ist der wichtigste Hook in React. Bei einer Änderung des Status wird die Komponente neu gerendert.
  • Wenn eine Komponente neu gerendert wird, dann werden auch alle darunterliegenden Komponenten neu gerendert.

useState und Rerendering

  • Das Rerendering einer Komponente passiert bei Statusänderungen in der Komponente selbst oder in einer ihrer Parent-Komponenten.
  • Das Rerendering passiert also häufiger und ist nicht kontrollierbar. Deswegen müssen Komponenten so geschrieben werden, dass sie bei jedem Rerendering korrekt funktionieren.
  • Dieses Verhalten wird auch als Pure Function bezeichnet. Eine Pure Function ist eine Funktion, die bei gleichen Eingabeparametern immer das gleiche Ergebnis liefert.
  • Der Zustand wird in useState gespeichert und bei einer Änderung des Zustands wird die Komponente neu gerendert.
  • Um sich werte zu merken, die nicht zu einem Rerendering führen, kann der useRef Hook verwendet werden.

  • Weitere Varianten von Touchable sind: TouchableHighlight, TouchableNativeFeedback, TouchableWithoutFeedback

Seiteneffekte und useEffect

  • Da eine Komponente bei jedem Rerendering neu gerendert wird, kann es sein, dass Seiteneffekte mehrfach ausgeführt werden. Deswegen müssen alle Seiteneffekte über den Hook useEffect ausgeführt werden.
  • useEffect wird nach dem Rerendering ausgeführt und hat folgende Signatur:
useEffect(() => {
    // Seiteneffekt
    return () => {
        // Cleanup
    }
}, [dependencies])
  • Der Cleanup wird ausgeführt, wenn die Komponente zerstört/unmounted wird.
  • Die dependencies sind ein Array von Werten, die mit den vorherrigen Werten verglichen werden. Wenn sich einer der Werte ändert, wird der Seiteneffekt erneut ausgeführt.
  • Wenn die dependencies leer sind, wird der Seiteneffekt nur einmalig beim ersten Rendering ausgeführt.
  • Wenn die dependencies nicht angegeben sind, wird der Seiteneffekt bei jedem Rerendering ausgeführt.

useEffect

Muss useEffect für jeden Seiteneffekt verwendet werden?

  • Ja, alle Seiteneffekte müssen über useEffect ausgeführt werden.
  • Mittels der Cleanup Funktion kann man z.B. Event-Listener entfernen, die vorher im useEffect hinzegefügt wurden.
  • Beispiel für das Hinzufügen und Entfernen eines Event-Listeners bei Reactnative:
useEffect(() => {
    const subscription = NetInfo.addEventListener(state => {
        console.log("Connection type", state.type);
        console.log("Is connected?", state.isConnected);
    });

    return () => {
        subscription();
    }
}, [])

Stale Closures und useEffect

  • Bei der Verwendung von useEffect kann es zu Problemen mit Stale Closures kommen.
  • Eine Stale Closure tritt auf, wenn eine Funktion auf eine Variable zugreift, die sich in einem anderen Scope befindet und sich seitdem geändert hat. Die Änderung wird nicht in der Funktion reflektiert.
  • Deswegen sollte die dependencies-Liste alle Variablen enthalten, auf die die Funktion zugreift. Dadurch führt React den Seiteneffekt erneut aus, wenn sich eine der Variablen ändert. Als Nebeneffekt wird die Stale Closure vermieden.

useCallback

  • useCallback ist ein Hook, der eine Callback-Funktion zurückgibt, die sich nicht bei jedem Rerendering ändert.
  • Wenn man eine Funktion in einer Komponente definiert, wird sie bei jedem Rerendering neu erstellt. Das kann zu Problemen führen, wenn die Funktion als Prop an eine Kindkomponente übergeben wird. Außerdem ist es zuweilen ineffizient, wenn die Funktion nicht von den Props abhängt.
const myfunc = useCallback(
    () => {
        doSomething(a, b)
    },
    [a, b]
)
  • useCallback gibt eine Funktion zurück, die sich nur ändert, wenn sich die dependencies ändern.
  • Der Typ der Funktion wird automatisch erkannt und durchgereicht. Somit ist der Typ des Callbacks der gleiche wie der der Funktion, die übergeben wird.

Übergreifende Zustände mittels Context

  • Zustände können über die Komponenten hinweg geteilt werden
  • Dazu wird der useContext Hook verwendet
  • Typischerweise wird ein Context in einer eigenen Datei definiert und mittels createContext erstellt und mit State kombiniert.
const MyContext = createContext(defaultValue);
export function MyProvider({children}) {
    const [state, setState] = useState(initialState);
    return (
        <MyContext.Provider value={{state, setState}}>
            {children}
        </MyContext.Provider>
    )
}

Verwendung eines Contexts

  • Auf der Ebene, wo der Zustand benötigt wird, wird der useContext Hook verwendet:
const {state, setState} = useContext(MyContext);
  • Gemäß der Rules of Hooks darf useContext nur in Funktionskomponenten oder in anderen Hooks verwendet werden.
  • Der State gehört zur Komponente in der er definitiert wurde (also dem Provider) und eine Änderung des States re-rendert auch alle Kindercomponenten des Providers.

  • Jeder Eintrag benötigt einen Schlüssel (Key), als Key eignet sich der Listenindex nur bedingt. Es ist besser einen Index zu verwenden, der den Eintrag wirklich eindeutig identifiziert.

type QuotesResponse = { quotes: Quote[], skip: number, limit: number }

export function Quotes() {

    useEffect(() => {
        (async () => {
            const response = await fetch('https://dummyjson.com/quotes')
            const json = await response.json() as QuotesResponse
            setQuotes(json.quotes)
        })() /* IFEE */
    }, [])
    return <ScrollView className="flex-1">
        {quotes.map(quote => <Text className="p-4" key={quote.id}>{quote.quote}</Text>)}
    </ScrollView>
}

FlatList, SectionList, VirtualizedList und ScrollView

  • FlatList ist eine Komponente, die eine Liste von Elementen rendert und dabei nur die Elemente rendert, die gerade sichtbar sind. Die Komponente basiert auf VirtualizedList und bietet eine bessere Performance als ScrollView. Die Performance ist nicht von der Anzahl der Elemente in der Liste abhängig, sondern von der Anzahl der Elemente, die gleichzeitig sichtbar sind.

  • SectionList ist eine Komponente, die eine Liste von Elementen rendert und dabei die Elemente in Abschnitte unterteilt. Jeder Abschnitt hat einen Titel und eine Liste von Elementen. Die Komponente basiert auf VirtualizedList und bietet eine bessere Performance als ScrollView.

  • VirtualizedList ist die Basis-Komponente für FlatList und SectionList. Die Komponente rendert nur die Elemente, die gerade sichtbar sind. Die Komponente bietet eine bessere Performance als ScrollView. Oftmals wird die Komponente nicht direkt verwendet, sondern FlatList oder SectionList verwendet.

  • ScrollView ist eine Komponente, die eine Liste von Elementen rendert und dabei alle Elemente rendert, auch die Elemente, die nicht sichtbar sind. Die Komponente bietet eine schlechtere Performance als FlatList, SectionList und VirtualizedList. Die Performance ist von der Anzahl der Elemente in der Liste abhängig.

Features und Attribute von FlatList

  • Eigenschaft data, um die Liste von Elementen zu steuern.
  • Eigenschaft renderItem, um die Komponente zu steuern, die für jedes Element gerendert wird.
  • Eigenschaft keyExtractor, um den Schlüssel für jedes Element zu steuern.
  • Eigenschaft onEndReached, um eine Funktion zu steuern, die aufgerufen wird, wenn das Ende der Liste erreicht wurde.
  • Eigenschaft onEndReachedThreshold, um den Schwellenwert für das Ende der Liste zu steuern.
  • Eigenschaft initialNumToRender, um die Anzahl der Elemente zu steuern, die beim ersten Rendern gerendert werden.
  • Eigenschaft ListHeaderComponent, um die Komponente zu steuern, die am Anfang der Liste gerendert wird.
  • Eigenschaft ListFooterComponent, um die Komponente zu steuern, die am Ende der Liste gerendert wird.
  • Eigenschaft ListEmptyComponent, um die Komponente zu steuern, die gerendert wird, wenn die Liste leer ist.
  • Eigenschaft refreshControl, um die Komponente zu steuern, die das Pull-to-Refresh-Feature implementiert.
  • Eigenschaft horizontal, um die Liste horizontal zu scrollen.

Prop-Drilling Problematik

  • Wenn Daten von einer Komponente an eine tief verschachtelte Komponente übergeben werden müssen, wird das Prop-Drilling Problem sichtbar, da via Props nur Daten von einer Komponente an die direkten Kinder übergeben werden können.

Responsive Gestaltung der App

  • Prinzipiell ist die Größe eines Containers gleich der Größe seines Inhalts plus Padding und Border

  • Die Margin zählt nicht zur Größe des Containers, wird aber bei der Darstellung vom Parent berücksichtigt.

  • Somit hat ein Element die Größe:

    • width = contentWidth + paddingLeft + paddingRight + borderLeft + borderRight
    • height = contentHeight + paddingTop + paddingBottom + borderTop + borderBottom

  • Die Größe eines Elements kann auch festgelegt werden, indem man width und height explizit setzt.

  • Die Größe eines Elements kann auch relativ zur Größe des Parent-Elements festgelegt werden.

  • Überschüssiger Inhalt wird standardmäßig gemäß seines Flex-Gewichts verteilt.

  • Bei fehlendem Platz wird der Inhalt ggf. gescrollt, abgeschnitten, umgebrochen oder verkleinert.

Größenabhängige Text-Elemente

  • Text-Elemente unterstützen die folgenden Props:
    • numberOfLines
    • ellipsizeMode
  • Wenn numberOfLines gesetzt ist, wird der Text nach der angegebenen Anzahl von Zeilen abgeschnitten. Die Art in der der Text abgeschnitten wird, kann mit ellipsizeMode festgelegt werden.
  1. head - Die Zeile wird so angezeigt, dass das Ende in den Container passt und der fehlende Text am Anfang der Zeile durch ein Auslassungszeichen angezeigt wird. z.B., “…wxyz”
  2. middle - Die Zeile wird so angezeigt, dass der Anfang und das Ende in den Container passen und der fehlende Text in der Mitte durch ein Auslassungszeichen angezeigt wird. “ab…yz”
  3. tail - Die Zeile wird so angezeigt, dass der Anfang in den Container passt und der fehlende Text am Ende der Zeile durch ein Auslassungszeichen angezeigt wird. z.B., “abcd…”
  4. clip - Zeilen werden nicht über den Rand des Textcontainers hinaus gezeichnet.

            const o = event.orientationInfo.orientation
            console.log('Orientation changed:', o);
        })
        return () => listener.remove()
    }, [])
    return (
        /* ... */
    )
}

Änderungen der Orientierung in einem Custom-Hook verfolgen

  • PS: Diese Funktionen können in einem Custom-Hook gekapselt werden.
import * as ScreenOrientation from 'expo-screen-orientation';
import { useEffect, useState } from 'react';

export function useOrientation() {
    const [orientation, setOrientation] = useState<ScreenOrientation.Orientation>(
            ScreenOrientation.Orientation.PORTRAIT_UP)
    useEffect(() => {
        const listener = ScreenOrientation.addOrientationChangeListener((event) => {
            const o = event.orientationInfo.orientation
            console.log('Orientation changed:', o);
        })
        return () => listener.remove()
    }, [])
    return orientation
}
  • PS: Der Hook useOrientation gibt die aktuelle Orientation zurück und verfolgt Änderungen der Orientation.
  • Der State in dem Hook wird auf die aktuelle Orientation gesetzt, wenn sich die Orientation ändert.
  • Der State bindet sich an die Komponente in der der Hook verwendet wird. (Generelles Prinzip von Hooks: Logik wird in Hooks ausgelagert und kann in Komponenten wiederverwendet werden)
  • Die Komponenten die den Hook verwenden, können die aktuelle Orientation abfragen und reagieren automatisch auf Änderungen der Orientation.

Einführung in Redux

  • Redux ist ein Zustandsmanager für JavaScript-Apps.
  • Redux ist neben Zustand einer der populärsten Zustandsmanager für React.

Alternativen zu Redux

  1. Zustand
  2. MobX
  3. useReducer und useContext (standard-mäßige built-in React Hooks)

Anlegen eines Redux-Stores

import { createStore } from 'redux'

const initialState = {
  counter: 0
}

const reducer = (state = initialState, action) => {
  switch (action.type) {
    case 'INCREMENT':
      return { counter: state.counter + 1 }
    case 'DECREMENT':
      return { counter: state.counter - 1 }
    default:
      return state
  }
}

const store = createStore(reducer)

Der Verwendung ist dann sehr einfach:

store.dispatch({ type: 'INCREMENT' })

Die Werte im Store sind reaktiv und können über store.getState() abgerufen werden.

store.subscribe(() => {
  console.log(store.getState())
})

Store-Daten selektieren

  • Um Daten aus dem Store zu selektieren, wird der Hook useSelector aus react-redux verwendet.

  • useSelector nimmt eine Funktion als Argument, die den Zustand des Stores als Argument erhält und die selektierten Daten zurückgibt.

  • Beispiel:

import {useSelector} from 'react-redux';
import {RootState} from './store';

HTTP-Zugriff mit der Fetch-API

  • Die fetch-API ist eine API um HTTP-Requests zu machen und in den Browser integriert. Auch bei Server-Side-Javascript kann die fetch-API verwendet werden.

  • Deswegen ist nur logisch dass React-Native auch die fetch-API verwendet.

  • Die fetch-API ist eine moderne Alternative zu ihren Vorgänger: XMLHttpRequest.

  • Die fetch-API ist asynchron und verwendet Promises.

Promises

  • Promises sind ein Konzept um asynchrone Operationen zu handhaben.

  • Ein Promise ist ein Objekt das einen Wert repräsentiert, der jetzt oder später verfügbar sein kann.

  • Um den Wert eines Promises zu erhalten, wird die Methode then() verwendet.

  • Mittels .then() registriert man eine Funktion/Callback die aufgerufen wird, wenn der Wert zur Verfügung steht. Dies wird auch als “Resolved-Promise” bezeichnet.

  • Ein Promise kann auch fehlschlagen. In diesem Fall wird die Methode catch() verwendet. Dies wird auch als “Rejected-Promise” bezeichnet.

Beispiel für die Verwendung der Fetch-API

fetch('https://dummyjson.com/quotes/1')
  .then(response => response.json())
  .then(json => console.log(json))

  • In diesem Beispiel wird ein HTTP-Request an die URL https://jsonplaceholder.typicode.com/todos/1 gemacht.

  • Der Request wird mit der Methode fetch() gemacht.

  • Die Methode fetch() gibt ein Promise zurück.

  • Das Promise wird mit der Methode then() aufgelöst.

  • Anschließend wird eine die Funktion response.json() aufgerufen die ein weiteres Promise zurückgibt.

  • Das zweite Promise wird mit der Methode then() aufgelöst.

  • Das Ergebnis wird in der Konsole ausgegeben.

Zweistufigkeit von Fetch

  • Die fetch-API ist zweistufig. Das bedeutet, dass die fetch-API zwei Promises zurückgibt.
  • Das erste Promise wird zurückgegeben, wenn der Request gemacht wird.
  • Das zweite Promise wird zurückgegeben, wenn die Antwort vom Server kommt.
  • Der Grund dafür ist, dass man bereits auf die Antwort der Header reagieren kann, bevor der Body der Antwort geladen wird. (So könnte man zum Beispiel aufgrund der Information im Header auf die weitere Verarbeitung der Antwort verzichten.)

Fetch-Optionen

  • Die Methode fetch() hat zwei Parameter: URL und Optionen.

  • Die Optionen sind ein Objekt mit verschiedenen Einstellungen für den Request.

    • method: HTTP-Methode (GET, POST, PUT, DELETE, etc.)
    • headers: Objekt mit HTTP-Headern
    • body: Daten die an den Server gesendet werden
    • mode: CORS-Modus (cors, no-cors, same-origin)
    • credentials: Cookies (omit, same-origin, include)
    • cache: Caching-Modus (default, no-store, reload, no-cache, force-cache, only-if-cached)

Beispiel für die Verwendung von Fetch-Optionen

fetch('https://jsonplaceholder.typicode.com/posts', {
  method: 'POST',
  headers: {
    'Content-Type': 'application/json'
  },
  body: JSON.stringify({
    title: 'foo',
    body: 'bar',
    userId: 1
  })
})
  .then(response => response.json())
  .then(json => console.log(json))
  • Hier wird eine Antwort im Format JSON angefordert.

ASync und Await allgemein

  • Promises sind eine gute Möglichkeit, asynchrone Operationen zu handhaben.

  • Es werden jedoch Funktionen registriert, welche mit den Werten der asynchronen Operationen aufgerufen werden sobald diese verfügbar sind. (Die .then() Methode)

  • Dies kann zu einer unübersichtlichen Verschachtelung von Funktionen führen und ist innerhalb von if-Bedingungen, for-Schleifen oder try-catch-Blöcken nicht immer ohne weiteres möglich.

  • Mit async und await können Promises in einer synchronen Art und Weise verwendet werden.

  • async wird vor einer Funktion deklariert, um zu signalisieren, dass die Funktion asynchron ist. Innerhalb der Funktion können dann Promises mit await “awaited” werden.

Vergleich von Promises und async/await

fetch('https://dummyjson.com/quotes/1')
  .then(response => response.json())
  .then(json => console.log(json))
async function fetchJson() {
  const response = await fetch('https://dummyjson.com/quotes/1')
  const json = await response.json()
  console.log(json)
}

- Tipp: Das Messen der Laufzeit mittels `console.time()` und `console.timeEnd()` ist mit `async` und `await` SEHR VIEL einfacher.

# Fehlerbehandlung mit async/await

- Fehlerbehandlung mit `async` und `await` ist einfacher als mit Promises.

- Mit `try` und `catch` können Fehler abgefangen werden.

```typescript
async function fetchJson() {
  try {
    const response = await fetch('https://dummyjson.com/quotes/1')
    const json = await response.json()
    console.log(json)
  } catch (error) {
    console.error(error)
  }
}
  • Und Fehler können auch sehr einfach weitergegeben werden, indem man sie schlichtweg nicht abfängt.

Zusammenfassung

  • Die fetch-API ist eine moderne Alternative zu XMLHttpRequest.
  • Die fetch-API ist asynchron und verwendet Promises.
  • Promises sind ein Konzept um asynchrone Operationen zu handhaben.
  • Mit async und await können Promises in einer synchronen Art und Weise verwendet werden.

Laden von Daten mittels Fetch-API, async und await, useEffect und useState

  • Um die Quotes von https://dummyjson.com/quotes zu laden, kann man folgenden TypeScript-Code verwenden:
import React, { useEffect, useState } from 'react';
import { ScrollView, Text } from 'react-native';

type Quote = { quote: string, author: string, id: number }
type QuotesResponse = { quotes: Quote[], skip: number, limit: number }

export function Quotes() {
    const [quotes, setQuotes] = useState<Quote[]>([])
    useEffect(() => {
        (async () => {
            const response = await fetch('https://dummyjson.com/quotes')
            const json = await response.json() as QuotesResponse
            setQuotes(json.quotes)
        })() /* IFEE */
    }, [])
    return <ScrollView className="flex-1">
        {quotes.map(quote => <Text className="p-4" key={quote.id}>{quote.quote}</Text>)}
    </ScrollView>
}

Erlöuterung des Ablaufs:

  • useState wird verwendet um den Zustand der Quotes zu speichern und mit einem leeren Array initialisiert.
  • useEffect wird verwendet um den Effekt des Ladens der Quotes zu hinterlegen. Der Effekt wird nur einmalig ausgeführt, wenn die Komponente gemounted wird. (Dependency-Array ist leer aber definiet)
  • Die Komponente Quotes rendert eine ScrollView mit den Quotes als Text-Komponenten. (Noch ist der Zustand leer, also wird nichts gerendert)
  • In der useEffect-Funktion wird ein IIFE (Immediately Invoked Function Expression) verwendet um asynchronen Code auszuführen, dies passiert nur einmalig, nachdem die Komponente gemounted wurde.
  • Im IIFE wird mittels fetch die Quotes von https://dummyjson.com/quotes geladen und dann mittels response.json() in ein JSON-Objekt umgewandelt.
  • Das JSON-Objekt wird dann mittels setQuotes in den Zustand der Komponente gespeichert.
  • Da der Zustand sich geändert hat, wird die Komponente neu gerendert und die Quotes werden angezeigt. Es wird also die gesamte Komponente neu ausgeführt. useState gibt nun den aktuellen Zustand, also die geladenen Quotes, zurück. useEffect wird erneut ausgeführt, aber da das Dependency-Array leer ist, wird der Effekt nicht erneut zur Ausführung hinterlegt.

Zusammenfassung

  • Es ist möglich, Daten mittels der Fetch-API zu laden.
  • Zusammen mit useEffect und useState können die geladenen Daten in einer React-Komponente angezeigt werden.
  • Mit async und await können Promises in einer synchronen Art und Weise verwendet werden.
  • Es existieren Alternativen zum Laden von Daten, wie z.B. Axios oder die Verwendung von Context und Reducern.
  • High-Level Bibiotheken wie Redux oder MobX können ebenfalls verwendet werden.
  • Eine sehr populäre Alternative ist die Bibliothek react-query (heutzutage Tanstack-Query) die das Laden, das Caching und die Verwaltung von Daten vereinfacht.

Asynchronizität und Fehlerbehandlung

  • Try/Catch kann auch mit async und await auf die gleiche Art und Weise verwendet werden wie mit Promises.

  • Fehler können mit throw geworfen werden.

  • Auch die Verwendung von finally ist möglich und funktioniert wie erwartet.

  • Bei der Verwendung von Promises ohne async und await ist es notwendig, die Fehlerbehandlung in jedem .then() zu machen und die Fehler weiterzugeben. Außerdem kann mit .catch() ein globaler Fehlerhandler definiert werden und mit .finally() kann ein Cleanup-Block registriert werden.

Pull-to-Refresh

  • Pull-to-refresh ist eine Gestensteuerung, die es ermöglicht, eine Liste zu aktualisieren wenn die Liste nach unten zieht.

  • React-Native bietet eine Komponente RefreshControl an, die es ermöglicht, Pull-to-Refresh zu implementieren.

  • Die RefreshControl-Komponente wird als refreshControl-Prop an die FlatList-Komponente übergeben.

  • Die RefreshControl-Komponente hat ein Attribut onRefresh, das eine Funktion erwartet, die aufgerufen wird, wenn die Liste aktualisiert wird.

  • Die RefreshControl-Komponente hat auch ein Attribut refreshing, das einen Boolean-Wert erwartet, der angibt, ob die Liste gerade aktualisiert wird.

  • Typischerweise wird refreshing auf true gesetzt, wenn die Liste aktualisiert wird, und auf false, wenn die Aktualisierung abgeschlossen ist.

  • FlatList, SectionList und ScrollView unterstützen alle die refreshControl-Prop.

Beispiel für die Verwendung von Pull-to-Refresh

import React, { useCallback, useEffect, useState } from 'react';
import { RefreshControl, ScrollView, Text } from 'react-native';

type Quote = { quote: string, author: string, id: number }
type QuotesResponse = { quotes: Quote[], skip: number, limit: number }

export function Quotes() {
    const [quotes, setQuotes] = useState<Quote[]>([])
    const [refreshing, setRefreshing] = useState(false)
    const refresh = useCallback(() => {
        (async () => {
            const response = await fetch('https://dummyjson.com/quotes')
            const json = await response.json() as QuotesResponse
            setQuotes(json.quotes)
        })() /* IFEE */
    }, [])
    useEffect(() => {
        refresh()
    }, [])
    return (
        <ScrollView className="flex-1"
            refreshControl={
                <RefreshControl refreshing={refreshing} onRefresh=refresh}
                    colors={['#2196F3']} // Android
                    tintColor="#2196F3" // iOS
                />}>
            { quotes.map(quote => <Text className="p-4" key={quote.id}>{quote.quote}</Text>) }
        </ScrollView >
    )
}

Authentifizierungsservice nutzen

  • Ein Login ist mittels OpenID Connect oder OAuth 2.0 möglich.

  • OAuth2 ist und OpenID Connect sind Protokolle, die es ermöglichen, sich bei einer Anwendung zu authentifizieren. OpenID ist eine Erweiterung von OAuth2 und ermöglicht es, sich bei einer Anwendung zu authentifizieren und gleichzeitig Zugriff auf Ressourcen zu erhalten.

  • Die Authentifizierung erfolgt über einen Authentifizierungsserver, der die Identität des Benutzers bestätigt. Der Server stellt einen Code aus, das die Identität des Benutzers bestätigt. Dieser Code wird auch Token genannt und wird vom Authentifizierungsserver mit einem Ablaufdatum versehen und vom Authentifizierungsserver signiert.

  • Mittels des öffentlichen Schlüssels des Authentifizierungsservers kann die Signatur des Tokens überprüft werden. Wenn die Signatur gültig ist, kann der Token entschlüsselt werden und die Identität des Benutzers wurde somit vom Authentifizierungsserver bestätigt. Man vertraut also dem Authentifizierungsserver, dass er die Identität des Benutzers korrekt bestätigt hat.

  • In der Regel gibt 2 es oder mehr Tokens.

    • Ein Access-Token, das den Zugriff auf Ressourcen erlaubt.
    • Ein Refresh-Token, das es ermöglicht, ein neues Access-Token zu erhalten, wenn das alte abgelaufen ist. Das Refresh-Token ist länger gültig als das Access-Token.
    • Ein ID-Token, das die Informationen zum Benutzer enthält (Anrede, Email, Rollen etc).

Verbreitete Authentifizierungsserver/-möglichkeiten

  • Keycloak ist eine open-source Identity and Access Management Lösung, die OAuth2 und OpenID Connect unterstützt. Keycloak kann lokal betrieben werden und wird von RedHat entwickelt. Die Verwendung von Keycloak ist kostenlos.

  • Auth0 ist ein Identity and Access Management Service, der OAuth2 und OpenID Connect unterstützt. Auth0 bietet eine Cloud-Lösung an, die einfach zu integrieren ist

  • Clerk ist vergleichbar mit Auth0 und ebenfalls ein Cloud-Server

  • Firebase Authentication ist ein Authentifizierungsservice von Google, der OAuth2 und OpenID Connect unterstütz

  • AWS Cognito ist ein Authentifizierungsservice von Amazon, der OAuth2 und OpenID Connect unterstützt

  • Microsoft EntraID (ehemals Azure Active Directory) ist ein Authentifizierungsservice von Microsoft, der SAML 2.0, OAuth2 und OpenID Connect unterstützt

Die Bereitstellung mit Expo

  • Expo biete EAS (Expo Application Services) an, für das Bauen und Veröffentlichen von React Native Apps.
  • EAS unterstützt iOS und Android, ist aber kostenpflichtig.

Deployment mit EAS

  • EAS ist sicherlich der einfachste Weg, eine React Native App zu veröffentlichen.
npx eas build --platform android
npx eas build --platform ios
npx eas submit --platform android
npx eas submit --platform ios
  • EAS baut die App und veröffentlicht sie in den App-Stores.
  • OTA (Over-the-Air) Updates sind ebenfalls möglich.

EAS Updates

  • Key points:
    • Aktuell noch möglich sowohl mit den aktuellen Policies des Apple App Stores und Google Play Stores
    • Updates nur der JS Codes, nicht des Native-Codes (Das kann ein Einschränkung sein, die für manche Updates nicht funktioniert, wenn zum Beispiel Module involviert sind, die auch native Code benötigen und dieser sich geändert hat)
  • Update auslöseN:
npx eas update --branch production --message "Bugfixes"
  • In der App
import * as Updates from 'expo-updates';

async function checkForUpdates() {
  try {
    const update = await Updates.checkForUpdateAsync()
    if (update.isAvailable) {
      await Updates.fetchUpdateAsync()
      await Updates.reloadAsync()
    }
  } catch (error) { /* Handle error */ }
}
  • Aktivierung von Updates in der app.json:
{
  "expo": {
   "updates": {
      "enabled": true,
      "fallbackToCacheTimeout": 0,
      "url": "https://u.expo.dev/your-project-id"
    },
  }
}
  • Und Update-Channels in der expo.json
{
  "cli": {
    "version": ">= 3.13.0"
  },
  "build": {
    "development": {
      "developmentClient": true,
      "distribution": "internal"
    },
    "preview": {
      "distribution": "internal",
      "channel": "preview"
    },
    "production": {
      "channel": "production"
    }
  },
  "submit": {
    "production": {}
  }
}
  • EAS Update wird von Expo gehostet, man muss also deren Service nutzen.
  • Wichtig: Diese Updates sind limitiert und funktionieren ohne Neu-Submit bei den App-Stores nicht für alle Änderungen. Insbesondere wenn sich der native Code ändert, muss die App neu gebaut und in den App-Stores veröffentlicht werden. (Also auch beim hinzufügen von neuen Modulen, die native Code benötigen)

Expo Pre-Builds

  • Expo kann auch ohne EAS verwendet werden.

  • Expo baut mittels npx expo prebuild die Anwendung für iOS und Android.

  • Natürlich wird der Build für iOS übersprungen wenn man keinen Mac benutzt

  • Der pre-build legt ein Android- und iOS Projekte an, welches in der jeweiligen IDE (Android Studio, XCode) geöffnet werden kann.

  • Für Android wird unterhalb des Verzeichnissen android ein entsprechendes Projekt mit gradle-Dateien und Kotlin-Sourcefiles angelegt.

  • Für iOS wird unterhalb des Verzeichnisses ios ein entsprechendes Projekt mit XCode-Projektdateien und Swift-Sourcefiles angelegt.

Icons und Startbildschirm

  • Icons und Startbildschirm können in der app.json konfiguriert werden.

  • Die Icons und der Startbildschirm werden in den Assets abgelegt.

  • Die Icons und der Startbildschirm werden in der app.json referenziert.

  • Achtung: Die Icons und der Startbildschirm müssen in verschiedenen Größen vorliegen.

  • Praktisch: Die Icons und der Startbildschirm können mit dem Expo CLI generiert werden.

  • Alternative auch mittels https://github.com/martiliones/icon-set-creator

Auflösung der Icons

  • Die Icons sollten in verschiedenen Auflösungen vorliegen.
    • Für iOS werden 3 Icons benötigt: 180x180, 120x120 und 60x60 Pixel.
    • Für Android werden 6 Icons benötigt: 192x192, 144x144, 96x96, 72x72, 48x48 und 36x36 Pixel.
  • Die Icons sollten im assets-Verzeichnis des Projekts abgelegt werden.
    • Die Icons sollten im PNG-Format vorliegen.
    • Die Icons sollten einen transparenten Hintergrund haben.

Der Startbildschirm (auch Splash Screen)

  • Der Startbildschirm sollte in verschiedenen Auflösungen vorliegen.

    • Für iOS werden 3 Icons benötigt: 1242x2688, 828x1792 und 1125x2436 Pixel.
    • Für Android werden 6 Icons benötigt: 1242x2688, 828x1792, 1125x2436, 1080x1920, 720x1280 und 480x800 Pixel.

  • Die Expo CLI kann den Startbildschirm generieren.

  • Der Startbildschirm sollte im assets-Verzeichnis des Projekts abgelegt werden.

    • Der Startbildschirm sollte im PNG-Format vorliegen.
    • Der Startbildschirm sollte einen transparenten Hintergrund haben.
    • Der Startbildschirm sollte das Logo der App enthalten.

Erstellung der Apps für iOS und Android

  • Die Erstellung der Apps für iOS und Android erfolgt über den Expo CLI.
  • Um die Apps zu erstellen, wird der Befehl expo build:ios bzw. expo build:android verwendet.
  • Dafür müssen die entsprechenden Zertifikate und Schlüssel hinterlegt sein.
  • Außerdem muss auch die Toolchain für die jeweilige Plattform installiert sein. Für MacOS ist dies Xcode, für Android ist dies Android Studio.
  • Android Studio basiert auf Java und IntelliJ IDEA und kann auch auf Linux, Windows und MacOS installiert werden. Für Xcode wird MacOS benötigt.

Erstellung der Apps in der Cloud mittels Expo Application Services EAS

  • Man kann die Apps auch über die Expo-Cloud erstellen lassen.
  • Dafür wird ein Account bei Expo Application Services EAS benötigt.
  • Auch das Hinterlegen der Zertifikate und Schlüssel ist notwendig. Der EAS kann diese auch automatisch generieren und die Apps auch in den App-Stores veröffentlichen.

Beispiel


# Erstellung der iOS-App
expo build:ios

# Erstellung der Android-App
expo build:android

MacOSX Buildfarms

  • Für die Erstellung der Apps in der Cloud stehen auch MacOSX Buildfarms zur Verfügung. Zum Beispiel bei Bitrise oder Codemagic.
